# 再生修正設計書（2026-07-04）

実使用フィードバックで確認した再生まわりの2つのバグを修正する。

## 問題1: 再生クロックが実時間より遅れる

`usePlaybackState` は `setInterval(16ms)` の各コールバックで「16ms 経過した」とみなして加算する。setInterval は正確に発火しない（負荷・省電力・バックグラウンドで間引かれる）ため、遅延分だけ再生が実 BPM より遅くなる。

**修正**: 実時計ベースに変更。tick ごとに `performance.now()` の前回差分を実経過時間として加算する（`speedMultiplier` は経過時間に乗算）。`play()` 開始時に基準時刻を取り、`pause()` でリセット。tick 間隔自体は表示更新頻度の意味しか持たなくなる。

## 問題2: 再生中のフォーカス小節が画面外に出る

`usePlaybackSync` は「進捗率 × スクロール可能高さ」の線形マッピングでスクロールする。コンテンツ高さは再生時間に比例しない（セクションごとの表示密度差、時間を消費しないタブ譜・歌詞のみセクション）ため、ハイライト小節が可視域から外れ、ヘッダ等の裏に隠れる。

**修正**: 進捗率ベースを廃止し、**ハイライト中の小節要素を直接追跡**する。

- 再生中に `currentMeasure` が変わるたび、スクロールコンテナ内の `.grid-measure.is-current-measure` を探し、その中心が**可視域の上から30%** の位置に来るよう `scrollTo({ behavior: 'smooth' })`（上端・プレイヤーバーからの余白を確保）
- 再生開始（再開）時も同様にスクロールし、現在小節へ復帰する
- DOM のハイライト更新後に要素を探す必要があるため、watch は `flush: 'post'`

## 挙動の整理（決定事項）

- **「一時停止中のスクロールでシーク」は廃止**（ユーザー確認済み）。スクロールは純粋に閲覧用。シークはシークバーと小節行タップ（既存）で行う
- 一時停止中のシークバー操作では自動スクロールしない（再生開始時に現在小節へスクロールされる）
- シークバーの `handleSeek` はシークのみ行い、スクロール副作用を持たない

## テスト方針

- `computeTargetScrollTop`（純関数）: アンカー位置計算と 0 クランプ
- 要素追跡: フェイクコンテナ/要素で「再生中の小節変化で scrollTo が呼ばれる」「停止中は呼ばれない」
- 実時計: フェイクタイマー（`toFake` に `performance` を含める）で既存テストを維持しつつ、経過時間ベースの加算を検証
- 実機確認: 縦長の曲＋歌詞のみセクション混在で、再生中ハイライトが常に可視域に留まること

- フォーカス小節のコード表示エリア（提案 (c)、別途設計）
